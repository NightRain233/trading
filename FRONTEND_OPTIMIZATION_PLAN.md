## ⚠️ 核心问题分析

### 1. “巨型组件”肿瘤 (The Giant Component Problem)
*   **现象**：`App.tsx` 超过 900 行，包含了从 Header 到列表、从排序逻辑到详情弹窗（ChartModal）的所有代码。
*   **后果**：任何一个小状态（如 Search 框打个字）的改变，都会触发 React 对整个超大组件树进行一次全量对比（Diffing），导致操作极其跟手感极差。

### 2. 主线程计算饱和 (Main Thread Blocking)
*   **现象**：21 个股票行的渲染、21 个迷你图的初始化、以及复杂的技术指标排序 logic 全部堆叠在主线程。
*   **后果**：用户点击按钮后，浏览器忙于处理 JS 计算，无法及时响应用户的物理输入。

### 3. 开发模式下的模块瀑布 (Vite Module Waterfall)
*   **现象**：在 `dev` 模式下，浏览器请求了数百个 `.tsx` 零碎模块。
*   **后果**：首屏发送 `watchlist` 请求前，有长达数秒的“空窗期”在等待代码下载和解析。

---

## 🛠 优化路线图 (Step-by-Step)

### 第一阶段：组件“碎块化”手术 (Component Extraction)
*   [x] **任务**：将 `App.tsx` 拆分为 `components/Header`、`components/FilterBar`、`components/Modals` 等独立模块。
*   **目的**：缩小单个文件的解析体积，提高 Vite 热更新效率，让代码结构符合现代开发规范。

### 第二阶段：精准控制渲染 (Render Optimization)
*   [x] **任务**：使用 `React.memo` 对股票行组件进行记忆。
*   **目的**：确保只有股价上涨/下跌的那一只 stock 重新渲染，其余股票保持静止，彻底消灭 1.1s 的输入延迟。

### 第三阶段：重量级模块“懒加载” (Lazy Loading & Suspense)
*   [x] **任务**：将包含 5 个图表计算逻辑的 `ChartModal` 改为 `React.lazy` 加载。
*   **目的**：首屏不再下载图表库代码，将首屏 JS 体积减少 40% 以上。

### 第四阶段：请求与渲染的“错峰出行”
*   [x] **任务**：初始化时优先渲染列表骨架，将迷你图（MiniChart）的渲染优先级调低。
*   [x] **任务**：实现“请求链条预热” (Request Pipeline Chaining)。
*   [x] **任务**：引入“连接复用” (Connection Multiplexing)——通过 Vite Proxy 转发所有 API 请求，消灭 127.0.0.1 在 macOS 下的初始连接延迟。
*   **目的**：让用户在 500ms 内先看到数字和文字，图表随后平滑滑入；同时彻底解决开发环境下后端重启导致的连接 hang 死问题。

*   **LCP** 降至 **1.5s** 以内。
